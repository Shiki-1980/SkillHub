# 讲演稿：DataHub 平台 AI Agent Skill 安全分析

> 15 分钟 · ~3,000 字

---

## 开场（1 分钟）

大家好。我今天报告的主题是 **DataHub 平台 AI Agent Skill 的安全分析**——简单说，就是如何用数据挖掘的方法，发现、攻击、并防御 AI Agent 技能市场中的恶意技能。

先看一组数字：OpenClaw 的 ClawHub 市场托管超过 13,000 个 community-contributed skill，安全审计发现 **13% 到 26% 包含安全漏洞**。2026 年初的 ClawHavoc 攻击活动，向市场推送了数百个恶意 skill。问题在于，现有的检测工具各管一段——正则扫描漏掉分段的 payload，静态分析看不懂自然语言里的 prompt injection。

我们做了什么？四个任务，形成 **检测→攻击→防御**的完整闭环。

---

## 数据与流水线概览（2 分钟）

首先说数据。我们从 DataHub Skills API 分两批次爬取了 **10,501 条** skill。每条 skill 包含 15 个字段：名称、描述、操作指令、权限声明、风险标签、分类等。核心是这四个文本字段——它们构成了我们后续所有分析的基础。

整个流水线是这样的：

```
数据爬取 → 弱标注 → 特征工程 → 异常检测（Task B）
                              → 对抗生成（Task C）
                              → 去风险化（Task D）
```

标注阶段我们用了双层策略：**8 条确定性规则**作为基础，加上 **DeepSeek-chat 的 LLM 辅助标注**。LLM 标注不是随便问"这个有没有害"，而是参考 SkillSieve 论文的四维分解——分别考察意图是否一致、权限是否合理、有没有隐蔽行为、最终综合判断。

最终标注分布：normal 79%，unsafe 20%，malicious 0.7%，useless 0.2%。

---

## 异常检测 — Task B（4 分钟）

异常检测我们做了三件事：**特征工程**、**无监督检测**、**有监督分类**。

**特征工程**用了三种表示：

- **TF-IDF**：500 维的 (1,2)-gram。为什么选 TF-IDF 而不是 BERT？Qiu 等人在 8 个日志数据集上证明，未微调的 BERT 并不优于 TF-IDF。
- **SBERT**：384 维的句子嵌入，用 all-MiniLM-L6-v2 模型。
- **结构化特征**：45 维，灵感来自 SkillSieve 的 Layer 1。包括风险标签计数、权限范围、危险关键词正则匹配等。

我们系统对比了 TF-IDF 和 SBERT。结果 SBERT 在全部三个检测器上胜出——Isolation Forest 提升 11%，LOF 提升 24%，XGBoost 提升 8%。这说明语义嵌入确实比词袋模型更好地编码了安全语义。

**无监督方法**我们用了三种：Isolation Forest、LOF、还有基于 Coote 等人论文的 LSTM Autoencoder。三个方法捕获了异常的不同维度——它们的共识只有 **45 条**，占总数的 0.8%。我们试了四种投票策略，"Any"策略效果最好，F1 达到 0.118，比最好的单独方法提升了 24%。

**有监督方面**，XGBoost 在 80/20 训练测试划分下，测试集 F1 达到 **0.648**。

**消融实验**是我们最有意思的发现。去掉 TF-IDF embedding 后，F1 反而上升了——这意味着对于安全分类任务，TF-IDF 的 500 维实际上是噪音，真正的信号在 45 维结构化特征里。去 LLM 标注也一样——因为规则标签和规则特征是同一套逻辑，模型几乎学到了确定性映射。

核心启示：**纯文本特征无法可靠捕获安全语义**。这也解释了为什么 SkillSieve 需要 Layer 2 的 LLM 语义分析。

## 对抗生成 — Task C（3 分钟）

检测系统的漏洞在哪儿？我们做了一个进化对抗生成框架，叫 **EASG**。

方法受三篇论文启发：AOP-Mal 的进化对抗恶意软件生成、GA Text Attack 的多目标遗传优化、ICLT 的 TF-IDF 引导扰动。

**6 个扰动算子**，全部基于确定性规则，完全可复现：

- O₁：把 TF-IDF 高权重词替换为风险更低的近义词，比如 exploit 变成 utilize
- O₂：插入良性填充句，稀释危险内容的密度
- O₃：把 `[danger]` 标签改成 `[warn]`，把权限描述模糊化
- O₄：从正常 skill 中借一段描述，注入到恶意 skill 里
- O₅：系统性地把 20 对风险词映射到中性词
- O₆：调整文本长度，匹配正常 skill 的统计分布

**遗传算法**：种群 50 个个体，进化 20 代。个体基因是算子序列，适应度有三项——逃逸率、语义相似度、结构相似度。用 Pareto 支配做多目标选择，两点交叉，精英保留。

结果很震撼：20 个种子 skill，初始平均被 1.95 个检测器标记，经过进化，**全部三个检测器都无法识别**，逃逸率 100%。平均语义相似度 0.75，也就是对抗样本仍然保留了原始的功能语义。

从算子贡献度来看，**结构模仿占了 39%**——这说明检测器最依赖的恰好是最脆弱的信号。你把文本长度调到正常范围，就能绕过很大一部分检测。

最关键的一点是：我们随后用这些对抗样本做了**对抗训练**。把对抗变体加入训练集，重训 XGBoost，在纯对抗样本上的 F1 从 **0 恢复到 0.49**，混合场景提升 0.42。这个闭环证明：攻击可以转化为防御能力。

---

## 去风险化 — Task D（3 分钟）

如果说 Task C 是红队攻击，Task D 就是蓝队防御。我们设计了一个 **KRI 复合风险评分模型**。

风险评分为什么是复合的？因为我们参考了一篇 2025 年的 KRI 论文——他们证明，CVSS 这种单一评分在真实漏洞数据上的 AUPRC 只有 0.011，接近随机猜测。而他们的三维模型达到了 0.927 的 ROC-AUC。

我们的 KRI 公式是：**0.55 × Threat + 0.25 × Impact + 0.20 × Exposure − SafetyBonus**。

这三个权重不是拍脑袋定的。我们做了网格搜索校准，发现 Threat 权重越高 F1 越好——操作威胁信号是 skill 安全评估中最有判别力的维度。最优配置把 F1 从 0.239 提升到 0.248。

**Safety Bonus** 是我们设计的一个创新机制。传统的安全评分只算"坏分数"，但我们认为真正的安全性还要看"好信号"——是否有用户确认机制？是否有审计日志？是否声明了最小权限原则？六种正向信号，每种降低 0.03 的风险，上限 0.15。

**7 个去风险算子**和 Task C 的逃逸算子形成精确对偶：

- Task C 把 `[danger]` 弱化成 `[warn]`，Task D 反过来把它改成 `[warn] + mitigated by user confirmation`
- Task C 注入假装合法的上下文，Task D 注入明确的风险声明
- Task C 模糊化命令，Task D 给每个危险命令打上"需要用户确认"的标记

在 80/20 的 holdout 测试集上，**84% 的 malicious/unsafe skill 成功降低了 KRI 评分**，平均降幅 0.052，语义相似度 0.93——去风险化几乎不损害功能描述。

## 可视化（1 分钟）

我们做了 6 张可视化图。最关键的是 TF-IDF 和 SBERT 嵌入空间的 t-SNE 对比。

TF-IDF 空间中，异常点散落在整个空间，和正常 skill 混在一起。SBERT 空间中，malicious 和 unsafe 形成了可辨识的孤立簇。

密度分析给了定量证据：在嵌入空间最稀疏的 10% 区域，SBERT 的异常浓度是 TF-IDF 的 1.7 倍。这直观验证了 SBERT 比 TF-IDF 更好地编码了安全语义。

## 总结与创新点（1 分钟）

四个任务串联起来形成完整闭环：10,501 条数据 → 4 种方法检测异常 → 进化算法生成对抗样本证明检测器脆弱性 → 去风险化框架提供防御方案 → 对抗训练将攻击转化为鲁棒性。

**5 个创新点**：

1. 将 SkillSieve 的三层检测框架适配到 DataHub 平台
2. EASG——首次将进化对抗生成应用于 AI agent skill 文本域
3. KRI Safety Bonus——在风险评分中引入正向安全信号
4. Task C↔D 对偶验证——同一个操作框架的两面
5. 攻击→防御闭环——对抗训练提升鲁棒性 0.49

**17 篇参考文献**，全部实际使用，覆盖检测、对抗、去风险三个领域的最新工作。

核心技术栈：Python + scikit-learn + XGBoost + PyTorch + sentence-transformers + DeepSeek API。
